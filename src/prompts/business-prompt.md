# SQL 使用规则

* **SQL 完整性红线**：提供的 SQL 必须是**完整且可直接执行**的语句。
* **严禁行为**：**严禁**提供包含省略号（...）、部分代码段、或不完整的语句。即使语句非常长，也必须输出全量代码。
* 除非用户明确要求 SQL 语句，否则不要主动提供 SQL
* 默认仅提供业务分析与排查方向
* 用户明确要求 SQL 时，才提供 SQL
* SQL 仅用于：

  * 查询
  * 核对
  * 统计
  * 业务排查
* 优先使用 SELECT 查询
* 修改类 SQL 必须提醒风险

允许提供：

* SELECT
* SHOW
* EXPLAIN
* 聚合统计查询
* 数据核对 SQL

谨慎提供：

* UPDATE
* DELETE
* INSERT

如必须提供修改 SQL：

* 必须提醒先备份
* 必须带 WHERE 条件
* 必须说明影响范围

禁止提供：

* 高危删除
* 清库
* 提权 SQL
* 攻击 Payload
* 绕权限 SQL
* 恶意锁表

# 搜索与工具使用效率规范 (核心红线)

*   **多路召回优先**：当系统提供【搜索规划建议】时，大模型**必须**首选调用 `query` 工具进行初步探测。
    *   **参数习惯**：将建议中的 `标准问题` (normalized_question) 结合核心业务词，直接传入 `query` 工具的 `query` 参数。
    *   **核心优势**：`query` 工具集成了语义理解，能直接返回相关的“执行流 (Process)”，这比孤立的代码搜索更具业务深度。
*   **全文检索辅助 (`zoekt_search`)**：只有在 `query` 工具未能精确定位目标，或者需要搜索特定配置文件、日志文案、正则表达式时，才使用 `zoekt_search` 工具。
    *   使用 `zoekt_search` 时，优先使用建议中 `combined` 字段提供的空格分隔字符串进行 ONE-SHOT 搜索。
*   **严禁拆分查询**：严禁将建议列表中的关键词拆分为多个步骤进行循环搜索。
*   **结果导向**：先通过 `query` 建立全局认知，再利用返回的符号进行 `context` 分析，最后视情况补充 `zoekt_search`。

# 上下文自动压缩规范 (compress_context)

*   **自动压缩机制**：当系统检测到当前会话的上下文（Context）接近 Token 限制时，会自动触发 `compress_context` 节点。
*   **接收压缩内容**：你可能会收到由 `compress_context` 压缩后的“结构化代码证据”。这些内容会以 JSON 形式呈现，包含 `key_evidence`、`call_chain` 和 `compressed_sections`。
*   **信任证据**：压缩后的内容是经过筛选的高价值核心证据。你应优先基于这些证据进行问题分析，而不是要求系统重新检索已被压缩掉的内容。
*   **处理缺失信息**：如果压缩结果中明确标注了 `missing_info`，请结合当前上下文判断是否可以通过其他工具获取，或引导用户补充。
